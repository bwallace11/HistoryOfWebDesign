XHTML 2.0 vs the emergence of HTML5

The web's fork in the road: purity vs pragmatism

In the early 2000s, the W3C focused on XHTML 2.0, a proposed successor to XHTML 1.x that broke compatibility with existing web content. At the same time, browser vendors grew frustrated that the official road map did not match real-world HTML usage.

This was the most important decision point in HTML's history. The web could have become a strict XML environment where existing pages stopped working, or it could evolve gradually while keeping backward compatibility. The pragmatic approach won.

Why this era replaced the previous one

HTML 4.01 and XHTML 1.x were stable but stagnant. The W3C stopped adding features that web developers needed for building applications (not just documents). Meanwhile, the web was becoming interactive: Ajax, Gmail, Google Maps. Authors needed richer form controls, multimedia elements, and application-level APIs. XHTML 2.0 did not address these needs and required authors to rewrite their sites in strict XML. Browser vendors decided to solve the problem themselves.

Artifact: The WHATWG Workshop (2004)

Title: Web Hypertext Application Technology Working Group
Year: 2004
Founders: Apple, Mozilla, and Opera representatives

At a W3C workshop in June 2004, a proposal to evolve HTML was voted down by W3C members who preferred the XHTML 2.0 direction. In response, engineers from Apple, Mozilla, and Opera formed WHATWG and began work on "Web Applications 1.0," which eventually became HTML5.

Why it mattered: WHATWG established the principle that HTML should evolve based on what browsers actually implement and what developers actually need, not on theoretical purity. This pragmatic approach defined the future of the web platform.

Artifact: The rise of web applications

Title: Gmail and Google Maps (Ajax applications)
Year: 2004-2005
Creator: Google

Gmail (2004) and Google Maps (2005) demonstrated that the web could deliver application-quality experiences. These products used XMLHttpRequest to load data without page refreshes, a technique Jesse James Garrett named "Ajax." They proved that HTML needed to evolve beyond a document markup language into an application platform.

Why it mattered: Ajax applications showed the gap between what developers were building and what the HTML specification supported. This gap motivated the creation of HTML5 with its application-focused APIs.

The XHTML 2.0 path

  • Removed familiar elements like img in favor of new XML-based patterns.
  • Required stricter XML rules, which many existing sites did not follow.
  • Ultimately abandoned; XHTML 2.0 never shipped in mainstream browsers.

The birth of WHATWG HTML5

  • In 2004, engineers from Apple, Mozilla, and Opera created the WHATWG (Web Hypertext Application Technology Working Group).
  • They started work on "Web Applications 1.0," which evolved into HTML5, focusing on backward compatibility and application features.
  • The W3C later restarted its HTML Working Group and collaborated with WHATWG on HTML5 drafts.

Limitations designers faced

  • No native audio or video elements; depended on Flash and other plugins.
  • No semantic elements for page structure; everything was a div.
  • Forms lacked validation, date pickers, and other common controls.
  • Different browsers still had different quirks modes and rendering behaviors.

Major contributors

  • Ian Hickson - principal editor of the HTML5 specification in WHATWG and W3C drafts.
  • Browser vendor representatives from Apple, Mozilla, Opera, Microsoft, and later Google - pushed for a practical, compatible HTML standard.
  • Jesse James Garrett - coined the term "Ajax" which described the new interactive web.
